# Meta-Build — Orchestrated Construction (Grok Native)

This is the high-discipline way to build things in the new surface.

It is not "just implement the feature." It is the full stack:

1. Correct context and priors (via prime / raven-prime)
2. Appropriate depth of thinking (MARS-style synthesis for complex or cross-domain work)
3. Safe, reviewable execution (calcinate-style contracts + verification gates where changes are risky)
4. Proper closure and distillation (so the learning compounds)

---

## When to Use This

- Any change that is more than a few lines or touches multiple concerns
- Work that has client, brand, security, or operational impact
- Situations where you want the full power of the OS instead of raw speed

For tiny things, just work. For anything that matters, use this flow.

---

## The Flow (Adapt to Scale)

**Light version** (most work):
- Prime the context
- Quick hypothesis + scoped plan
- Implement with surgical discipline (Karpathy rules from AGENTS.md)
- Verify
- Distill the key learning

**Heavy version** (complex, cross-domain, high-stakes):
- Prime → load correct lane + topology/brand priors
- MARS-style decomposition and parallel discovery if the problem has real systems depth
- Detailed plan with explicit verification criteria
- Implementation using verifiable surgery patterns (calcinate) where appropriate
- Full 4-check or equivalent gates if production-adjacent (Raven side)
- Distill into reusable skill / update to doctrine / client brief as needed

---

## The Non-Negotiables

- Never skip context loading on anything non-trivial.
- Never ship changes without verification that matches the risk level.
- Always distill at the end of significant work. The OS only gets better if you feed it.

This flow, when internalized, is what makes the difference between "using Grok" and "operating inside a true power-user Reality OS on Grok."

Use it. It will feel slow at first and then dramatically faster and safer than the old way.
